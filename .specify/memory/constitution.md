## 목적 및 비전

CryptoTitan 블로그는 수익화에 초점을 맞춘 암호화폐 투자 블로그 플랫폼으로, 깔끔한 Medium 스타일 인터페이스를 통해 프리미엄 교육 콘텐츠를 제공합니다. 이 플랫폼은 구독 기반 프리미엄 콘텐츠, 뉴스레터, 그리고 향후 제휴 파트너십을 통해 수익을 창출하며, 높은 편집 기준과 사용자 신뢰를 유지합니다.

**핵심 미션:** 독자들이 기꺼이 비용을 지불할 만한 탁월한 가치를 제공함으로써 암호화폐 투자 지식을 지속 가능한 수익으로 전환합니다.

## 핵심 원칙

### 원칙 1: 프리미엄 콘텐츠 보호

**프리미엄 콘텐츠는 반드시 기술적으로 보호되고 접근이 제어되어야 합니다.**

- `isPremium: true`로 표시된 모든 게시물은 렌더링 전에 유효한 활성 구독 인증을 요구해야 함
- 미들웨어는 콘텐츠 전달 전 모든 `/blog/[slug]` 라우트에서 구독 확인을 강제해야 함
- API 응답은 비구독자에게 프리미엄 콘텐츠를 미리보기, 발췌문, 메타데이터로 유출해서는 안 됨
- 프리미엄 상태는 서버 측에서 검증되어야 하며, 클라이언트 측 확인만으로는 불충분함
- 구독 상태는 Stripe 웹훅 이벤트와 실시간으로 동기화되어야 함

**근거:** 보호가 강제되지 않으면 핵심 수익화 모델이 실패합니다. 프리미엄 콘텐츠에 대한 무료 접근은 수익원을 제거하고 구독 제안의 가치를 떨어뜨립니다.

**현재 상태:** ❌ 미구현 - 출시 전 중요 차단 요소

### 원칙 2: 디자인 일관성 및 Medium 스타일 미학

**모든 사용자 대면 인터페이스는 Medium의 깔끔하고 미니멀한 디자인 언어를 따라야 합니다.**

- 타이포그래피: 제목은 세리프 폰트(예: Georgia, Charter), 본문은 산세리프
- 색상 팔레트: 주로 검정(#000), 회색(#757575), 흰색(#FFF); 최소한의 강조 색상
- 레이아웃: 콘텐츠 우선, 넉넉한 여백, 그림자/그라디언트 대신 테두리 구분선
- 게시물 카드: 기본 변형은 오른쪽 정렬된 작은 이미지와 가로 레이아웃
- 버튼: 단순한 둥근 또는 알약형 모양에 단색 배경; 그라디언트 금지
- 네비게이션: 미니멀하고 눈에 띄지 않는 헤더; 무거운 배경 없는 깔끔한 푸터

**근거:** Medium의 디자인은 읽기 집중도와 콘텐츠 참여를 극대화하는 것으로 입증되었습니다. 일관성은 유료 구독에 필수적인 신뢰와 전문성을 구축합니다.

**현재 상태:** ⚠️ 부분 구현 - 홈페이지와 PostCard 업데이트 완료; Header, Footer, 카테고리 페이지, 블로그 상세 페이지 업데이트 필요

### 원칙 3: 콘텐츠 가치 및 편집 품질

**모든 콘텐츠는 실행 가능하고 충분히 조사된 투자 인사이트를 제공해야 합니다.**

- 프리미엄 콘텐츠는 깊이, 독점성 또는 적시성으로 페이월을 정당화해야 함
- 모든 투자 조언은 위험 공시 및 면책 조항을 포함해야 함
- 출처를 인용해야 하며, 추측성 의견은 명확히 표시해야 함
- 콘텐츠는 정의된 6개 카테고리에 올바르게 분류되어야 함:
  - 입문자 라운지
  - 실전 투자관
  - 고급 전략실 (DeFi, NFT)
  - 인사이트 라운지
  - 실패 투자 아카이브
  - Titan 생태계 이야기 (개인 브랜딩)
- 출판 전 교정 및 팩트체크가 이루어져야 함

**근거:** 콘텐츠 품질은 구독 유지율 및 브랜드 평판과 직접적으로 연관됩니다. 낮은 품질의 콘텐츠는 해지와 부정적인 입소문으로 이어집니다.

### 원칙 4: 수익화 윤리 및 투명성

**수익 창출은 절대 사용자 신뢰나 편집 독립성을 훼손해서는 안 됩니다.**

- 제휴 링크는 "제휴 링크" 또는 "Affiliate" 라벨로 명확히 공개되어야 함
- 스폰서 콘텐츠는 "Sponsored" 또는 "협찬"으로 눈에 띄게 표시되어야 함
- 가격은 투명해야 하며, 숨겨진 수수료나 예상치 못한 요금이 없어야 함
- 구독 취소는 쉽고 명확하게 접근 가능해야 함
- 환불 정책(7일, 무조건)은 준수되고 공개적으로 명시되어야 함
- 개인적인 금전적 인센티브가 투자 추천에 영향을 미쳐서는 안 됨

**근거:** 신뢰는 구독 비즈니스의 기초입니다. 기만적인 관행은 단기 수익을 창출하지만 장기 생존 가능성을 파괴합니다.

**현재 상태:** ✅ 구현됨 - Stripe 통합 투명; 구현 시 제휴 공개 시스템 필요

### 원칙 5: 사용자 경험 우선

**모든 기능은 읽기 및 발견 경험을 향상시켜야 하며, 방해해서는 안 됩니다.**

- 페이지 로드 시간은 4G 연결에서 3초 미만이어야 함
- 모바일 경험은 일류여야 하며, 반응형 디자인은 필수
- 네비게이션은 직관적이어야 하며, 사용자는 2번의 클릭 내에 콘텐츠를 찾을 수 있어야 함
- 양식은 명확한 검증 피드백과 오류 메시지를 제공해야 함
- 로딩 상태는 전달되어야 함 (스피너, 스켈레톤, 진행 표시기)
- 접근성은 최소 WCAG 2.1 AA 표준을 충족해야 함

**근거:** 나쁜 UX는 이탈을 유발합니다. 독자들은 무료 대안이 존재할 때 느리고 혼란스럽거나 답답한 경험을 용납하지 않습니다.

**현재 상태:** ⚠️ 부분 구현 - 로딩 상태, 모바일 메뉴, 접근성 감사 누락

### 원칙 6: 보안 및 개인정보 보호

**사용자 데이터는 업계 표준 보안 관행으로 보호되어야 합니다.**

- 인증은 안전한 세션 관리(JWT 또는 데이터베이스)와 함께 NextAuth를 사용해야 함
- 자격증명 인증 사용 시 비밀번호는 bcrypt(최소 10라운드)로 해시되어야 함
- API 라우트는 XSS 및 SQL 인젝션을 방지하기 위해 모든 사용자 입력을 검증하고 정제해야 함
- 민감한 환경 변수(API 키, 시크릿)는 절대 버전 관리에 커밋되어서는 안 됨
- 프로덕션에서는 HTTPS가 강제되어야 함 (Vercel이 자동 처리)
- 사용자 데이터는 명시적 동의 없이 제3자에게 판매되거나 공유되어서는 안 됨
- GDPR/CCPA 준수는 개인정보 보호 정책 및 데이터 내보내기 기능으로 해결되어야 함

**근거:** 보안 침해는 즉시 신뢰를 파괴하고 법적 책임을 노출합니다. 개인정보 침해는 막대한 벌금과 영구적인 브랜드 손상을 초래할 수 있습니다.

**현재 상태:** ⚠️ 부분 구현 - 입력 검증 강화, 속도 제한, 법적 페이지 필요

### 원칙 7: 성능 및 최적화

**플랫폼은 기능을 희생하지 않고 빠르고 효율적인 경험을 제공해야 합니다.**

- 이미지는 적절한 크기 조정 및 지연 로딩과 함께 Next.js Image 컴포넌트를 사용해야 함
- API 응답은 적절한 경우 캐시되어야 함 (Sanity CDN, SWR/React Query)
- 데이터베이스 쿼리는 최적화되어야 함 (적절한 인덱스, 선택적 필드 가져오기)
- 번들 크기는 모니터링되어야 하며, 큰 종속성에는 코드 분할이 사용되어야 함
- Lighthouse 성능 점수는 프로덕션 빌드에서 ≥90이어야 함

**근거:** 성능은 SEO 순위, 전환율, 사용자 만족도에 직접적인 영향을 미칩니다. 느린 사이트는 순위가 낮고 전환율이 나쁩니다.

**현재 상태:** ✅ 구현됨 - Next.js 14 최적화 적용; 지속적인 모니터링 필요

### 원칙 8: SEO 및 검색 가능성

**콘텐츠는 유기적 검색을 통해 검색 가능해야 고객 획득 비용을 줄일 수 있습니다.**

- 모든 페이지는 고유하고 설명적인 메타 제목과 설명을 가져야 함
- 기사 및 저자에 대한 구조화된 데이터(JSON-LD)가 구현되어야 함
- Sitemap.xml이 생성되고 검색 엔진에 제출되어야 함
- 중복 콘텐츠 페널티를 피하기 위해 정규 URL이 올바르게 설정되어야 함
- 소셜 공유를 위한 Open Graph 및 Twitter Card 메타 태그가 있어야 함
- Core Web Vitals는 "양호" 임계값을 충족해야 함 (LCP < 2.5s, FID < 100ms, CLS < 0.1)

**근거:** 유료 광고는 비쌉니다. 유기적 검색은 지속 가능하고 저비용의 고객 획득을 제공합니다. SEO는 장기적인 경쟁 우위입니다.

**현재 상태:** ❌ 미구현 - 성장에 중요

## 거버넌스

### 수정 절차

1. **제안:** 모든 팀원은 이 파일에 대한 풀 리퀘스트를 통해 수정을 제안할 수 있음
2. **검토:** 수정은 프로젝트 소유자의 검토 및 승인이 필요함
3. **버전 상향:**
   - MAJOR: 후방 호환 불가능한 원칙 제거 또는 재정의
   - MINOR: 새 원칙 추가 또는 실질적 확장
   - PATCH: 명확화, 표현 개선, 오타 수정
4. **전파:** 병합 후 48시간 내에 모든 종속 템플릿 및 문서 업데이트

### 준수 및 검토

- **출시 전:** 모든 중요 원칙(1, 6, 8)은 프로덕션 배포 전에 구현되어야 함
- **주간:** 원칙과의 정렬을 보장하기 위해 미해결 이슈 검토
- **월간:** 원칙 2-7 준수 감사; 예외 및 개선 계획 문서화
- **분기별:** 포괄적인 헌장 검토; 원칙이 더 이상 프로젝트 목표에 부합하지 않으면 수정 제안

### 집행

- 원칙을 위반하는 풀 리퀘스트는 수정될 때까지 차단되어야 함
- 원칙 준수를 깨는 프로덕션 배포는 즉시 롤백되어야 함
- 원칙을 위반하는 기술 부채는 개선을 위해 우선순위가 지정되어야 함

## 구현 우선순위 (현재 상태 기반)

### 1단계: 중요 차단 요소 (출시 전)

1. **원칙 1 (프리미엄 보호):** 미들웨어 및 구독 검증 구현
2. **원칙 6 (보안):** 인증 시스템 완료, 입력 검증 추가
3. **원칙 8 (SEO):** 메타 태그, 사이트맵, 구조화된 데이터 추가

### 2단계: 사용자 경험 (출시 주간)

1. **원칙 2 (디자인):** Header, Footer, 카테고리 페이지, 블로그 상세 페이지의 Medium 스타일 업데이트 완료
2. **원칙 5 (UX):** 로딩 상태, 모바일 메뉴, 오류 처리 추가

### 3단계: 성장 및 최적화 (출시 후)

1. **원칙 4 (수익화):** 적절한 공개와 함께 제휴 마케팅 구현
2. **원칙 3 (콘텐츠):** 편집 일정 및 품질 검토 프로세스 수립
3. **원칙 7 (성능):** 지속적인 모니터링 및 최적화
